# ADR-001: 302 (Found) over 301 (Moved Permanently) for redirects

**Status:** accepted
**Date:** 2026-07-14

## Context
A short-link redirect can answer 301 or 302. Browsers and intermediaries cache
301 aggressively (often indefinitely); after the first visit, subsequent clicks
never reach our server.

## Decision
We answer **302**. Every click hits link-service, which preserves click
analytics (the promoted-listing aggregation pipeline in `analytics-stream`
consumes these events in Phase 6) and lets a link's target be changed or
disabled after creation.

## Alternatives considered
- **301** — lowest server load (CDN/browser absorbs repeat clicks), but
  analytics-blind and irrevocable once cached. Right choice only if the mapping
  is truly immutable and click data is worthless.
- **308/307** — method-preserving variants; irrelevant for GET-only redirects.

## Consequences
- Every click costs us a request → the read path must be cheap: cache-aside
  Redis in front of Postgres (measured in `load/NUMBERS.md`).
- If read load ever dwarfs analytics value, flipping to 301 is a one-line
  change that trades observability for offload — the classic exam answer.
